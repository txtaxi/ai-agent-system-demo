# 数据库设计精读

## 文档目标

这份文档专门讲数据库，不再混杂前端、接口和工作流。

目标是让你理解：

- 这个项目为什么要设计这些表
- 各表之间是什么关系
- 每张表在系统里承担什么职责
- 数据库设计是如何服务于 RAG、Agent、反馈和 HITL 的

你读完这份文档，应该能回答：

`为什么这个项目不是只有 documents 和 messages 两张表，而是需要 knowledge_bases、document_chunks、agent_runs、feedback 等一整套结构。`

---

## 1. 先从整体看数据库的职责

这个项目的数据库不是只负责“保存聊天记录”。

它同时承担了 4 层职责：

1. `知识层`
   - 知识库
   - 文档
   - 文档分块

2. `对话层`
   - 会话
   - 消息

3. `Agent 执行层`
   - 运行记录
   - 工作流轨迹
   - 审核状态

4. `评测层`
   - 用户反馈
   - 反馈统计来源

所以数据库设计不是围绕“聊天”展开的，而是围绕：

`知识 -> 问答 -> 执行 -> 反馈`

这条完整闭环展开的。

## 3. `knowledge_bases`：为什么需要知识库表

文件：
[knowledge_base.py](D:\Study\Agent\StudyPlan\enterprise-ai-agent-system\backend\app\models\knowledge_base.py)

### 它表示什么

一个知识库实体。

例如：

- 人事制度库
- 销售制度库
- 产品知识库

### 为什么不能直接把文档全放在一起

因为企业知识不是平铺的。

不同知识往往有：

- 不同业务边界
- 不同检索目标
- 不同权限边界
- 不同问答上下文

如果没有 `knowledge_bases` 这层，后面就很难做：

- 自动选库
- 跨库检索
- 知识库级统计
- 企业级权限隔离

### 它和谁关联

- 一个 `knowledge_base`
  - 有多个 `documents`

所以它是知识层的顶层容器。

## 4. `documents`：为什么需要文档表

文件：
[document.py](D:\Study\Agent\StudyPlan\enterprise-ai-agent-system\backend\app\models\document.py)

### 它表示什么

一份原始上传文档。

### 它一般存什么

- 属于哪个知识库
- 原始文件名
- 文件类型
- 存储路径
- 当前处理状态
- 错误信息

### 为什么不能只保留文件本身

因为系统不仅要“有文件”，还要知道：

- 文件有没有处理成功
- 当前是什么状态
- 出错时为什么失败
- 它属于哪个知识库

### 状态字段为什么重要

文档在系统里不是静态对象，而是有生命周期：

- `uploaded`
- `processing`
- `processed`
- `failed`

这个状态流转让前端和后端都能知道：

`这份文档目前能不能被问答使用`

### 它和谁关联

- 属于一个 `knowledge_base`
- 对应多个 `document_chunks`

## 5. `document_chunks`：为什么这是 RAG 最核心的表之一

文件：
[document_chunk.py](D:\Study\Agent\StudyPlan\enterprise-ai-agent-system\backend\app\models\document_chunk.py)

### 它表示什么

被切分后的文档片段。

### 它一般存什么

- 所属 `document_id`
- `chunk_index`
- `content`
- `metadata_json`
- `embedding`

### 为什么不能直接对整篇 document 检索

因为向量检索的最小单位通常不是整篇文档，而是片段。

原因是：

- 整篇太长，语义混杂
- 检索精度差
- 引用不可控
- 回答时上下文难组织

### 为什么要存 `chunk_index`

因为后续需要：

- 保持原始顺序
- 组织相邻上下文
- 展示引用片段位置

### 为什么要存 `metadata_json`

因为不同 chunk 除文本外，还可能需要携带：

- 来源文件名
- 文件类型
- 分块策略
- 标题信息
- 段落信息

这让后续：

- 引用展示
- 负反馈分析
- 调试分块策略

更容易做。

### 为什么 embedding 直接存在这里

因为检索时最常用的对象就是 chunk。

把向量直接和 chunk 绑定，检索链最简单：

`query -> embedding -> chunk vector distance`

## 6. `conversations`：为什么需要会话表

文件：
[conversation.py](D:\Study\Agent\StudyPlan\enterprise-ai-agent-system\backend\app\models\conversation.py)

### 它表示什么

一段连续对话的容器。

### 为什么不能只有 messages

因为消息必须有上层归属。

如果没有 `conversations`：

- 历史消息很难分组
- 会话列表无法存在
- 前端无法切换不同会话
- 运行记录也很难和某段对话对应

### 它在系统里的作用

它是“聊天体验”的顶层容器。

一个 conversation 下挂多条 message，也可能关联多条 agent_run。

## 7. `messages`：为什么要单独存消息

文件：
[message.py](D:\Study\Agent\StudyPlan\enterprise-ai-agent-system\backend\app\models\message.py)

### 它表示什么

会话中的一条消息。

包括：

- 用户提问
- 系统回答

### 为什么它不是简单字符串列表

因为消息除了文本，还会伴随：

- 角色
- 引用
- 时间
- 所属会话

这些都需要落库。

### 它在系统里的价值

消息表让系统支持：

- 历史消息回放
- 多轮对话
- 对话页重进恢复
- 审核通过后补写系统回答

## 8. `agent_runs`：为什么这是 Agent 工程里最关键的表之一

文件：
[agent_run.py](D:\Study\Agent\StudyPlan\enterprise-ai-agent-system\backend\app\models\agent_run.py)

### 它表示什么

一次 Agent 执行。

你可以把它理解成：

`一条问题从进入工作流到产出结果的完整执行记录`

### 为什么不能只看最终 message

因为最终消息只能告诉你：

- 说了什么

但不能告诉你：

- 为什么这么说
- 选了什么 route
- 用了什么工具
- 检索质量如何
- 是否触发人工审核

### 它一般存什么

- 所属 `conversation_id`
- 用户原始问题
- route
- 状态
- `trace_json`

### `trace_json` 为什么重要

因为很多 Agent 中间状态结构是动态变化的，不适合每个字段都拆成数据库列。

所以这里用 `trace_json` 保存：

- 工作流步骤
- 摘要
- 决策原因
- 中间状态

这让系统既能保留过程信息，又不会把表结构膨胀得过于复杂。

### `agent_runs` 的核心意义

这张表使系统具备：

- 可观测性
- 可追溯性
- 失败复盘
- HITL 审核承载能力

这是普通聊天系统和 Agent 工程系统的重要区别之一。

## 9. `feedback`：为什么反馈表这么重要

文件：
[feedback.py](D:\Study\Agent\StudyPlan\enterprise-ai-agent-system\backend\app\models\feedback.py)

### 它表示什么

用户对某次运行结果的评价。

### 它通常存什么

- `agent_run_id`
- 反馈值（有帮助 / 没帮助）
- 备注
- 时间

### 为什么反馈要绑定 `agent_run_id`

因为如果只绑定消息文本或 conversation，会导致后续分析很模糊。

绑定到 run 的好处是：

你可以明确知道：

- 这次差评对应哪次执行
- 它走了什么 route
- 它选了什么 tool
- 它命中了哪些知识库
- 它是不是需要人工审核

### 它的工程意义

这张表让项目拥有了最小评测闭环：

- 使用
- 反馈
- 聚合
- 失败样本分析

## 10. 各表之间的关系

最关键的关系链如下：

### 知识链

- `knowledge_bases`
  - 1 对多 `documents`
  - `documents`
    - 1 对多 `document_chunks`

### 对话链

- `conversations`
  - 1 对多 `messages`
  - 1 对多 `agent_runs`

### Agent 链

- `agent_runs`
  - 多对 1 `conversations`
  - 1 对多 `feedback`

如果画成简化版，可以理解为：

```text
knowledge_bases -> documents -> document_chunks
conversations -> messages
conversations -> agent_runs -> feedback
```

## 11. 为什么数据库设计能支撑 Agent 而不仅是聊天

如果你只做聊天产品，数据库可能只需要：

- users
- conversations
- messages

但这个项目之所以需要更多表，是因为它不只是聊天。

它还要支撑：

### 1. 知识管理

所以需要：

- `knowledge_bases`
- `documents`
- `document_chunks`

### 2. 执行过程记录

所以需要：

- `agent_runs`

### 3. 反馈闭环

所以需要：

- `feedback`

### 4. HITL

虽然当前没有独立审核表，但 `agent_runs` 已经承载了待审核和审核结果状态。

也就是说：

`数据库设计其实直接反映了这个系统的能力边界`

## 12. 为什么有些信息存列，有些信息存 JSON

这是数据库设计里很值得学的一点。

这个项目采用的是：

- 稳定核心字段：单独列
- 动态过程信息：JSON

### 为什么知识库名、文档状态这些放列

因为它们：

- 稳定
- 常筛选
- 常排序
- 常展示

### 为什么 trace 放 JSON

因为工作流步骤、摘要、分析信息经常演化。

如果把它们全拆成列，会导致：

- 表经常改结构
- 迁移频繁
- 灵活性差

所以这里采用：

`核心结构化字段 + 动态 JSON 细节`

这是一种很常见的工程折中。

## 13. 这个数据库设计未来怎么扩到企业级

如果后面要继续做企业级，最可能新增的是：

- `users`
- `roles`
- `permissions`
- `audit_logs`
- 更独立的 `review_tasks`

但当前结构已经很好扩展，因为：

- 知识层独立
- 对话层独立
- Agent 层独立
- Eval 层独立

也就是说后面更多是“新增模块”，不是“推翻重来”。
