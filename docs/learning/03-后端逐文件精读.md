# 后端逐文件精读

## 文档目标

这份文档不是简单列文件名，而是按“如何真正读懂后端”的方式，解释：

- 每个目录为什么存在
- 每个核心文件解决什么问题
- 代码之间是怎么串起来的
- 你应该先看什么，再看什么

如果你把这份文档读完，再对照代码看一遍，基本就能把整个后端主链路掌握住。

---

## 1. 后端整体结构

后端目录在：

[backend](D:\Study\Agent\StudyPlan\enterprise-ai-agent-system\backend)

核心结构是：

```text
backend/
  app/
    main.py
    api/
    core/
    db/
    models/
    schemas/
    services/
    rag/
    agent/
  requirements.txt
  Dockerfile
```

你可以把它理解成 7 层：

1. `main.py`
应用入口

2. `api/`
接口层

3. `core/`
配置层

4. `db/`
数据库基础设施层

5. `models/`
数据库模型层

6. `schemas/`
接口数据结构层

7. `services/ + rag/ + agent/`
业务能力层

## 2. 从入口开始读

## 2.1 `backend/app/main.py`

文件：
[main.py](D:\Study\Agent\StudyPlan\enterprise-ai-agent-system\backend\app\main.py)

这是整个后端的入口。

它主要做 4 件事：

1. 创建 FastAPI 应用
2. 注册 CORS
3. 启动时初始化 pgvector 扩展和建表
4. 挂载全部 API 路由

为什么这个文件很重要：

因为它告诉你：

- 应用是怎么启动的
- 数据库初始化在什么时候发生
- 所有接口是通过哪个总路由挂进去的

读完它，你就知道整个后端的“壳”是什么样。

---

## 2.2 `backend/app/api/router.py`

文件：
[router.py](D:\Study\Agent\StudyPlan\enterprise-ai-agent-system\backend\app\api\router.py)

这是 API 总路由表。

它把下面这些模块统一挂载进来：

- `health`
- `knowledge_bases`
- `documents`
- `retrieval`
- `chat`
- `conversations`
- `agent_runs`
- `feedback`

为什么这个文件重要：

因为它像“后端目录导航页”。

你可以根据它反向知道：

- 系统有哪些业务模块
- 接口是按什么领域拆分的

## 3. 配置层和数据库层

## 3.1 `backend/app/core/config.py`

文件：
[config.py](D:\Study\Agent\StudyPlan\enterprise-ai-agent-system\backend\app\core\config.py)

这是配置中心。

它负责统一读取：

- 应用名
- 版本
- API 前缀
- PostgreSQL 连接信息
- Redis 地址
- 上传目录
- OpenAI 兼容接口配置
- 模型名
- embedding 维度

为什么要有它：

因为没有配置中心的话，数据库地址、模型地址、目录路径就会散落在各个文件里，后面非常难维护。

---

## 3.2 `backend/app/db/session.py`

文件：
[session.py](D:\Study\Agent\StudyPlan\enterprise-ai-agent-system\backend\app\db\session.py)

它负责创建：

- `engine`
- `SessionLocal`

这里的意义是：

- `engine` 负责数据库连接池
- `SessionLocal` 负责每次请求实际使用的数据库会话

这是 SQLAlchemy 项目里非常典型的一层。

---

## 3.3 `backend/app/db/base.py`

文件：
[base.py](D:\Study\Agent\StudyPlan\enterprise-ai-agent-system\backend\app\db\base.py)

这个文件的作用通常是导出统一的 `Base`，让模型定义和建表逻辑有统一入口。

它本身不复杂，但它是模型层能正常工作的基础设施之一。

## 4. 模型层：数据库里到底存什么

模型目录：
[models](D:\Study\Agent\StudyPlan\enterprise-ai-agent-system\backend\app\models)

## 4.1 `knowledge_base.py`

文件：
[knowledge_base.py](D:\Study\Agent\StudyPlan\enterprise-ai-agent-system\backend\app\models\knowledge_base.py)

这个表表示一个知识库。

例如：

- 人事制度库
- 销售制度库
- 产品知识库

它是整个系统知识边界的顶层容器。

---

## 4.2 `document.py`

文件：
[document.py](D:\Study\Agent\StudyPlan\enterprise-ai-agent-system\backend\app\models\document.py)

这个表表示上传进来的一个原始文档。

它一般会存：

- 属于哪个知识库
- 文件名
- 文件类型
- 存储路径
- 当前状态
- 错误信息

这个表解决的问题是：

`系统需要知道“原始文件”是什么，以及它处理到哪一步了`

---

## 4.3 `document_chunk.py`

文件：
[document_chunk.py](D:\Study\Agent\StudyPlan\enterprise-ai-agent-system\backend\app\models\document_chunk.py)

这是 RAG 的核心表之一。

每个文档被切分后，会形成很多 chunk。

每个 chunk 一般包含：

- 对应哪个 document
- chunk 顺序
- 文本内容
- metadata
- embedding 向量

这个表解决的问题是：

`检索并不是对原始整篇文档做的，而是对文档片段做的`

---

## 4.4 `conversation.py`

文件：
[conversation.py](D:\Study\Agent\StudyPlan\enterprise-ai-agent-system\backend\app\models\conversation.py)

表示一个会话。

它解决的问题是：

`多轮对话不能只靠前端缓存，必须落库`

---

## 4.5 `message.py`

文件：
[message.py](D:\Study\Agent\StudyPlan\enterprise-ai-agent-system\backend\app\models\message.py)

表示会话中的每条消息。

包括：

- 用户提问
- 系统回答

这个表让系统支持：

- 会话列表
- 历史消息恢复
- 多轮对话展示

---

## 4.6 `agent_run.py`

文件：
[agent_run.py](D:\Study\Agent\StudyPlan\enterprise-ai-agent-system\backend\app\models\agent_run.py)

这是 Agent 工程里非常关键的一张表。

它表示：

`一次智能体执行`

里面保存的不是最终答案本身，而是执行过程的记录，例如：

- 问题是什么
- 走了什么 route
- 选了什么工具
- 是否进行了重排
- 是否需要人工审核
- trace_json 里记录了哪些步骤

这张表是可观测性的核心。

---

## 4.7 `feedback.py`

文件：
[feedback.py](D:\Study\Agent\StudyPlan\enterprise-ai-agent-system\backend\app\models\feedback.py)

这个表保存用户对一次 run 的反馈。

例如：

- 有帮助
- 没帮助
- 备注

它的意义是：

`第一次把“模型表现”变成可统计数据`

---

## 4.8 `mixins.py`

文件：
[mixins.py](D:\Study\Agent\StudyPlan\enterprise-ai-agent-system\backend\app\models\mixins.py)

这里一般放公共字段混入，例如：

- `id`
- `created_at`
- `updated_at`

这样每个模型就不用重复写一遍。

## 5. Schema 层：接口的数据边界

目录：
[schemas](D:\Study\Agent\StudyPlan\enterprise-ai-agent-system\backend\app\schemas)

这层的重点不是业务逻辑，而是：

`定义接口输入输出长什么样`

## 5.1 `knowledge_base.py`

定义知识库相关的请求和响应结构。

## 5.2 `document.py`

定义：

- 上传文档响应
- 文档列表项
- 文档处理结果
- chunk 返回结构

## 5.3 `retrieval.py`

定义检索接口的请求和响应。

包括：

- query
- knowledge_base_id / knowledge_base_ids
- 返回的 chunk 列表

## 5.4 `chat.py`

这是问答接口最重要的 schema 之一。

它定义：

- 问答请求
- 问答响应
- 引用结构
- 命中知识库结构

随着项目进化，这个文件持续扩展过很多次，例如后来加了：

- `selected_tool`
- `selected_tool_reason`
- `human_review_required`
- `awaiting_human_review`
- `agent_run_id`

这说明 schema 层是系统能力演化的一个直接窗口。

## 5.5 `agent_run.py`

定义运行记录详情的返回结构。

后来这个文件不断加字段，就是为了让运行记录页能展示更多执行信息。

## 5.6 `feedback.py`

定义反馈提交、反馈统计、负反馈样本返回结构。

## 6. Service 层：真正的业务核心

目录：
[services](D:\Study\Agent\StudyPlan\enterprise-ai-agent-system\backend\app\services)

如果你只能花时间精读一层，我建议优先读这里。

## 6.1 `knowledge_base_service.py`

知识库服务。

负责知识库的基本增删查逻辑。

它不复杂，但它把数据库操作从路由层拿了出来。

---

## 6.2 `document_service.py`

文件：
[document_service.py](D:\Study\Agent\StudyPlan\enterprise-ai-agent-system\backend\app\services\document_service.py)

这是知识库数据流的关键文件之一。

它负责两类事：

### 第一类：上传阶段

- 校验知识库是否存在
- 生成存储文件名
- 把文件保存到磁盘
- 在数据库创建 `document`

### 第二类：处理阶段

- 根据文件类型读取文本
- 支持 `txt / md / pdf / docx`
- 调用 `chunk_document`
- 对每个 chunk 生成 embedding
- 写入 `document_chunks`
- 更新文档状态

这个文件的重要性很高，因为它把“原始文件”变成了“可检索知识”。

---

## 6.3 `embedding_service.py`

负责生成向量。

这个项目里 embedding 是一个独立服务层，而不是写死在检索逻辑里，这样后面替换模型更容易。

---

## 6.4 `retrieval_service.py`

文件：
[retrieval_service.py](D:\Study\Agent\StudyPlan\enterprise-ai-agent-system\backend\app\services\retrieval_service.py)

它负责：

- 把 query 变成向量
- 通过 pgvector 做相似度查询
- 把数据库结果包装成检索结果对象

这个文件只解决一件事：

`给你候选内容`

它不管：

- route
- tool
- answer

这就是单一职责。

---

## 6.5 `knowledge_base_router_service.py`

这是后面从“用户手选知识库”升级到“系统自动选库”的关键文件。

它负责根据问题选择：

- 一个知识库
- 或多个候选知识库

这是整个系统变得更像真实产品的重要一步。

---

## 6.6 `query_router_service.py`

这个文件负责问题分类。

把问题分为：

- `qa`
- `summary`
- `comparison`

这一步让系统不再只有一条问答路径。

---

## 6.7 `llm_service.py`

文件：
[llm_service.py](D:\Study\Agent\StudyPlan\enterprise-ai-agent-system\backend\app\services\llm_service.py)

这个文件是“模型相关能力”的集中管理层。

它主要负责：

- 最终回答生成
- 工具选择
- 查询改写
- 改写评估

而且它采用的是：

`规则优先，模型辅助`

这很符合工程上的保守策略：

- 模型可用时增强能力
- 模型不可用时仍然可以回退

这个文件在项目中非常重要，因为它决定了：

`模型不是随意到处调用，而是被服务层统一封装`

---

## 6.8 `conversation_service.py`

负责：

- 获取 / 创建会话
- 保存消息

这个文件让 `chat_service.py` 不必直接写数据库细节。

---

## 6.9 `chat_service.py`

文件：
[chat_service.py](D:\Study\Agent\StudyPlan\enterprise-ai-agent-system\backend\app\services\chat_service.py)

这是问答主流程的核心协调文件。

它做的事很多，但本质上可以概括成：

1. 准备会话
2. 创建 `agent_run`
3. 保存用户消息
4. 执行 LangGraph 工作流
5. 组装响应
6. 如果需要人工审核，就进入待审核
7. 否则保存系统回答
8. 更新运行记录

流式问答也是在这里做的，只是返回阶段换成了 SSE。

这个文件的角色不是“做所有智能逻辑”，而是：

`协调各层`

---

## 6.10 `agent_run_service.py`

运行记录服务。

负责：

- 创建 run
- 完成 run
- 标记失败
- 标记待审核
- 审核通过
- 审核驳回

这是 Agent 可观测性和 HITL 的核心服务之一。

---

## 6.11 `feedback_service.py`

反馈服务。

负责：

- 保存反馈
- 聚合统计
- 返回负反馈样本

后期项目演化时，这个文件不断增强，从简单记录发展成了 Eval 聚合中心。

---

## 7. RAG 层：文档切分是怎么做的

目录：
[rag/chunking](D:\Study\Agent\StudyPlan\enterprise-ai-agent-system\backend\app\rag\chunking)

这里很值得学习，因为它体现了项目从“简单分块”逐步升级的过程。

## 7.1 `base.py`

定义 chunker 的抽象接口。

## 7.2 `paragraph_chunker.py`

普通段落优先的切分方式。

## 7.3 `section_chunker.py`

适合标题/章节结构的切分。

## 7.4 `policy_chunker.py`

适合制度、条款、编号类文档。

## 7.5 `router.py`

分块路由器。

它的意义是：

`不是所有文档都应该用同一种切法`

这也是项目从简单 RAG 向更成熟设计演化的重要一环。

## 8. Agent 层：系统为什么已经不是普通 RAG

目录：
[agent](D:\Study\Agent\StudyPlan\enterprise-ai-agent-system\backend\app\agent)

## 8.1 `tools.py`

这个文件把工作流里会用到的能力封装成“工具”。

典型工具有：

- 检索知识
- 总结上下文
- 对比上下文
- 构建 QA 上下文
- 构建引用
- 分析检索质量
- 判断是否需要更多上下文
- 重排结果
- 判断是否需要人工审核

这一步很重要，因为它把系统从：

`写死的流程代码`

推进成：

`可复用的能力单元`

## 8.2 `workflow.py`

文件：
[workflow.py](D:\Study\Agent\StudyPlan\enterprise-ai-agent-system\backend\app\agent\workflow.py)

这是整个 Agent 的大脑。

它定义了：

- `AgentState`
- 每个节点函数
- 节点之间的连线
- 条件分支

当前核心节点大致包括：

- `route_question`
- `rewrite_query`
- `assess_rewrite`
- `tool_select`
- `search_knowledge`
- `analyze_retrieval`
- `decide_more_context`
- `expand_search`
- `rerank_results`
- `assess_human_review`
- `prepare_*_context`
- `generate_answer`

为什么这个文件最值得精读：

因为它是这个项目里“Agent”概念最具体的体现。
