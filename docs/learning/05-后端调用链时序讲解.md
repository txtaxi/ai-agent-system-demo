# 后端调用链时序讲解

## 文档目标

前面的文档已经讲清了：

- 每个文件做什么
- 项目怎么一步步演化过来

这份文档专门讲：

`一次请求进入后端之后，到底按什么顺序流过整个系统`

也就是你常说的：

`调用链`

我会用最常见的几条主链路来讲：

1. 创建知识库
2. 上传并处理文档
3. 发起一次问答
4. 查看运行记录
5. 提交反馈并做统计
6. 人工审核通过 / 驳回

---

## 1. 先记住后端调用链的总原则

在这个项目里，后端基本遵循这条路线：

`HTTP 请求 -> API 路由 -> Schema 校验 -> Service 处理 -> Model 持久化 -> 返回响应`

如果是问答链路，则会变成：

`HTTP 请求 -> API 路由 -> chat_service -> LangGraph workflow -> tools/services -> 持久化 -> 返回响应`

所以你理解调用链时，先记住 4 层：

1. `routes`
接口边界

2. `schemas`
数据结构边界

3. `services`
业务逻辑

4. `models`
数据库状态

而 Agent 链路是在 `services` 里面通过 `workflow.py` 串起来的。

入口文件：
[backend/app/api/routes/knowledge_bases.py](D:\Study\Agent\StudyPlan\enterprise-ai-agent-system\backend\app\api\routes\knowledge_bases.py)

主要调用链：

- 是典型 CRUD

## 3. 上传并处理文档调用链

这条链路是知识库系统真正开始有价值的地方。

### 3.1 上传文档

入口文件：
[backend/app/api/routes/documents.py](D:\Study\Agent\StudyPlan\enterprise-ai-agent-system\backend\app\api\routes\documents.py)

核心服务：
[backend/app/services/document_service.py](D:\Study\Agent\StudyPlan\enterprise-ai-agent-system\backend\app\services\document_service.py)

调用链如下：

1. 前端发起 `POST /api/v1/documents/upload`
2. `documents.py` 接收文件和 `knowledge_base_id`
3. 路由层调用 `save_uploaded_document`
4. `document_service.py`
   - 校验知识库是否存在
   - 创建上传目录
   - 生成唯一文件名
   - 把文件写入磁盘
   - 创建 `Document` 记录
5. 返回文档元数据

到这里，文件只是“上传成功”，还没有变成可检索知识。

### 3.2 处理文档

入口仍然在：
[backend/app/api/routes/documents.py](D:\Study\Agent\StudyPlan\enterprise-ai-agent-system\backend\app\api\routes\documents.py)

对应接口：

- `POST /api/v1/documents/{document_id}/process`

调用链如下：

1. 路由层拿到 `document_id`
2. 调用 `process_document`
3. `document_service.py`：
   - 查出 `Document`
   - 状态改为 `processing`
   - 根据文件类型读取文本
   - 调用 `chunk_document`
   - 对每个 chunk 调用 `generate_embedding`
   - 写入 `DocumentChunk`
   - 状态改为 `processed`
4. 返回处理结果

这里最关键的中间调用是：

- [backend/app/rag/chunking/router.py](D:\Study\Agent\StudyPlan\enterprise-ai-agent-system\backend\app\rag\chunking\router.py)
- [backend/app/services/embedding_service.py](D:\Study\Agent\StudyPlan\enterprise-ai-agent-system\backend\app\services\embedding_service.py)

这一条链把：

`原始文件`

变成：

`可检索的向量分块`

## 4. 发起一次问答调用链

这是整个项目最重要的一条调用链。

入口文件：
[backend/app/api/routes/chat.py](D:\Study\Agent\StudyPlan\enterprise-ai-agent-system\backend\app\api\routes\chat.py)

核心服务：
[backend/app/services/chat_service.py](D:\Study\Agent\StudyPlan\enterprise-ai-agent-system\backend\app\services\chat_service.py)

工作流文件：
[backend/app/agent/workflow.py](D:\Study\Agent\StudyPlan\enterprise-ai-agent-system\backend\app\agent\workflow.py)

### 4.1 非流式问答主链

调用顺序如下：

1. 前端调用 `POST /api/v1/chat/ask`
2. `chat.py` 接收 `ChatAskRequest`
3. 路由层把数据库会话和 payload 交给 `ask_with_rag`
4. `chat_service.py`
   - 获取或创建会话
   - 创建一条 `agent_run`
   - 保存用户消息
   - 调用 `run_agent_workflow`
5. `workflow.py` 开始执行 LangGraph 图
6. 图执行完成后，`chat_service.py`
   - 用 `compose_response_from_state` 组装响应
   - 用 `build_trace_payload` 组装轨迹
   - 如果命中人工审核，标记 `pending_review`
   - 否则保存系统消息
   - 更新 `agent_run`
7. 返回 `ChatAskResponse`

### 4.2 工作流内部调用顺序

工作流不是一条直线，它是多步状态机。

核心顺序大致如下：

1. `route_question`
   - 调用 `classify_query`
   - 判断是 `qa / summary / comparison`

2. `rewrite_query`
   - 调用 `llm_service.rewrite_query`

3. `assess_rewrite`
   - 调用 `llm_service.assess_query_rewrite`
   - 判断这次改写值不值得参与检索

4. `tool_select`
   - 调用 `llm_service.select_tool`
   - 决定后续该走哪个上下文构建工具

5. `search_knowledge`
   - 调用 `agent/tools.py` 里的 `search_knowledge`
   - 它内部又会调：
     - 知识库路由服务
     - 检索服务

6. `analyze_retrieval`
   - 评估检索质量

7. `decide_more_context`
   - 判断是否需要继续补充搜索

8. `expand_search`
   - 若需要，则扩大检索范围再查一轮

9. `rerank_results`
   - 对候选结果按任务类型再重排

10. `assess_human_review`
   - 判断是否需要进入人工审核

11. `prepare_*_context`
   - `qa` 走 `build_qa_context`
   - `summary` 走 `summarize_context`
   - `comparison` 走 `compare_contexts`

12. `generate_answer`
   - 调用 `llm_service.generate_answer`

## 5. 流式问答调用链

还是从：
[backend/app/api/routes/chat.py](D:\Study\Agent\StudyPlan\enterprise-ai-agent-system\backend\app\api\routes\chat.py)

进入，不过这次走的是：

- `POST /api/v1/chat/ask/stream`

调用链：

1. 路由层调用 `stream_ask_with_rag`
2. `stream_ask_with_rag` 内部先复用 `ask_with_rag`
3. 普通问答执行完成后，按片段切分回答
4. 通过 `_sse_event` 包装成 `text/event-stream`
5. 前端逐步接收 `meta / chunk / done`

这条链说明：

`流式输出不是另一套问答逻辑，而是另一种返回方式`

## 6. 查看运行记录调用链

入口：
[backend/app/api/routes/agent_runs.py](D:\Study\Agent\StudyPlan\enterprise-ai-agent-system\backend\app\api\routes\agent_runs.py)

服务：
[backend/app/services/agent_run_service.py](D:\Study\Agent\StudyPlan\enterprise-ai-agent-system\backend\app\services\agent_run_service.py)

调用链：

1. 前端调用 `GET /api/v1/agent-runs`
2. 路由层调用 run 查询服务
3. service 从数据库拿出 `agent_runs`
4. schema 把它们包装成前端可显示的结构
5. 返回列表

查看单条详情时：

1. 前端调用 `GET /api/v1/agent-runs/{id}`
2. 路由层取单条 run
3. 从 `trace_json` 中拿出摘要和步骤
4. 返回详情响应

这条链的重点不是复杂，而是理解：

`运行记录页展示的数据，本质上来自 agent_run + trace_json`

## 7. 提交反馈与获取反馈统计调用链

入口：
[backend/app/api/routes/feedback.py](D:\Study\Agent\StudyPlan\enterprise-ai-agent-system\backend\app\api\routes\feedback.py)

服务：
[backend/app/services/feedback_service.py](D:\Study\Agent\StudyPlan\enterprise-ai-agent-system\backend\app\services\feedback_service.py)

### 7.1 提交反馈

调用链：

1. 前端在聊天页面点击“有帮助 / 没帮助”
2. 调用 `POST /api/v1/feedback`
3. 路由层接收 `agent_run_id`
4. `feedback_service.py` 创建 `Feedback`
5. 数据落库

### 7.2 获取统计

调用链：

1. 前端打开反馈统计页
2. 调用 `GET /api/v1/feedback/stats`
3. `feedback_service.py`：
   - 查询 feedback
   - 关联 agent_run
   - 从 `trace_json.summary` 中提取 route、tool、retrieval_quality、knowledge_base 等信息
   - 做聚合统计
4. 返回统计结果

### 7.3 获取负反馈样本

调用链：

1. 前端打开负反馈样本区
2. 调用 `GET /api/v1/feedback/negative-samples`
3. service 查询负反馈并组织摘要
4. 返回给前端列表展示

这条链很值得学，因为它体现了：

`可观测性数据如何被反向用于质量分析`

## 8. 人工审核通过 / 驳回调用链

入口：
[backend/app/api/routes/agent_runs.py](D:\Study\Agent\StudyPlan\enterprise-ai-agent-system\backend\app\api\routes\agent_runs.py)

服务：
[backend/app/services/agent_run_service.py](D:\Study\Agent\StudyPlan\enterprise-ai-agent-system\backend\app\services\agent_run_service.py)

调用链：

1. 某次问答命中高风险，工作流把 run 标记成 `pending_review`
2. 前端在运行记录页看到该 run
3. 用户点击“通过”或“驳回”
4. 前端调用 agent_runs 审核接口
5. `agent_run_service.py`
   - 更新 run 状态
   - 写入审核结果
   - 如果通过，补写系统回答到会话
   - 如果驳回，记录驳回原因

这条链说明系统已经具备了：

`真正的中断 / 恢复`

## 9. 一次完整问答的超简时序总结

如果你只想记最核心的一条链，可以记成：

1. 前端发问题
2. `chat.py` 收请求
3. `chat_service.py` 准备会话和 run
4. `workflow.py` 执行 Agent 图
5. `tools.py + services/*` 提供检索、判断、重排、上下文整理能力
6. `llm_service.py` 负责改写、工具选择、回答生成
7. `chat_service.py` 保存消息和运行记录
8. 路由层返回响应给前端

如果你要用一句话记住后端调用链，可以记成：
