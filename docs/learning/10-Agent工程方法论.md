# Agent 工程方法论

## 这份文档解决什么问题

前面的文档已经回答了：

- 这个项目做了什么
- 它是怎么一步一步进化的

这份文档进一步回答：

- 为什么真实的 Agent 项目通常要这样推进
- 每一步在工程上到底意味着什么
- 以后你自己做项目时，应该按什么思路拆阶段

它更偏“方法论”，而不是“功能清单”。

## 一、为什么 Agent 项目不能一开始就追最终形态

很多人一开始就会想做：

- 多工具智能体
- 多步骤工作流
- LangGraph + ReAct + 评测

问题在于，如果底座不稳，这些能力会变成概念堆叠。

Agent 项目通常是多层耦合系统：

```text
知识输入
  ->
分块
  ->
向量化
  ->
检索
  ->
上下文组织
  ->
回答生成
  ->
路由 / 工具选择
  ->
工作流
  ->
记录与评测
```

如果前面一层不稳，后面所有层都会被污染。

所以真实开发更常见的方式是：

`先打通主链路，再逐层增加智能度和可观测性`

## 二、为什么这个项目的推进顺序是合理的

这个项目采用的顺序是：

```text
工程骨架
  ->
知识库
  ->
文档处理
  ->
分块与向量化
  ->
RAG 问答
  ->
会话与流式
  ->
任务路由
  ->
自动选库
  ->
Agent Tools
  ->
LangGraph
  ->
运行记录
  ->
大模型辅助工具选择
```

这个顺序合理，是因为每一步都在解决一个真实工程问题。

### 1. 先做工程骨架

没有：

- 前后端结构
- 数据库连接
- Docker
- 文档体系

后面所有功能都只是散点代码。

### 2. 先做知识输入

RAG 和 Agent 都依赖真实知识。

如果文档还没有真正进入系统，后面的：

- 检索
- 问答
- 路由
- 工作流

都没有真实数据支撑。

### 3. 先做 RAG，再做 Agent

Agent 不是凭空工作的，它通常建立在：

- 搜索
- 检索
- 工具
- 规则

这些能力之上。

所以：

- 没有稳定 RAG，就很难做出稳定 Agent

### 4. 先做路由，再做工作流

工作流不是画图就有价值。

它成立的前提是：

- 系统已经存在不同任务类型
- 不同任务类型已经开始走不同处理路径

先有：

- `qa`
- `summary`
- `comparison`

再接 LangGraph，图才有意义。

### 5. 先做功能，再做可观测性

没有功能时，监控和追踪价值不大。

但当系统开始进入：

- 自动选库
- 工具选择
- LangGraph

这种黑盒感明显增强的阶段，就必须补：

- 运行记录
- 节点轨迹
- 可追溯性

## 三、这个项目体现了哪些真实工程原则

### 原则 1：先打通，再优化

例如分块策略的演化：

- 第一版：固定窗口
- 第二版：多策略分块

这不是前面白做，而是典型工程节奏：

- 先证明主链可行
- 再在真实问题暴露后做结构升级

### 原则 2：先功能闭环，再抽象结构

例如：

- 一开始问答逻辑主要在服务层
- 后来抽成 Agent Tools
- 再后面接 LangGraph

这说明抽象不是越早越好，而是：

`当你已经知道哪些能力稳定存在，再做抽象`

### 原则 3：阻塞问题优先修

真实开发里不是所有问题都立刻深修。

更合理的是：

- 阻塞主链路的问题先修
- 不阻塞的问题先记录
- 继续推进主线

### 原则 4：逐步让系统可解释

很多 AI 项目一开始只追求“能跑”，但越做越难查。

这个项目后面补 `agent_runs`，本质上就是把系统从：

- 可运行

推进到：

- 可追溯
- 可解释
- 可调试

## 四、为什么这个项目适合用来学习 Agent 工程

因为它不是一开始就站在“最终 Agent”上，而是一步一步长出来的。

它会逼你学会真正值钱的东西：

### 1. 学会区分底座能力和智能体能力

底座能力包括：

- 后端服务
- 数据库
- 向量库
- 文档处理
- 接口
- 容器化

智能体能力包括：

- 路由
- 工具选择
- 工作流
- 运行记录

真实 Agent 项目需要两者一起成立。

### 2. 学会阶段化交付

每个阶段都能形成一个相对完整的版本。

这很重要，因为真实团队里，项目也很少是最后一天一次性上线。

### 3. 学会从“结果问题”定位到“系统层问题”

例如一句“回答不准”，背后可能来自：

- 文档处理
- 分块
- 检索
- 路由
- 模型生成

只有真的做过分层系统，才会建立这种定位意识。

## 五、如果你以后自己从零做一个 Agent 项目，推荐顺序是什么

### 第一步：先搭工程骨架

- 后端框架
- 前端框架
- 数据库
- Docker
- 文档体系

### 第二步：做知识输入

- 知识库
- 文档上传
- 文档处理
- 分块
- 向量化

### 第三步：做最小 RAG

- 检索
- 问答
- 引用

### 第四步：做产品感

- 会话
- 历史消息
- 流式输出

### 第五步：做智能度

- 任务路由
- 自动选库
- 不同任务不同处理流程

### 第六步：做 Agent 化

- Agent Tools
- LangGraph
- 工具选择节点

### 第七步：做可观测性

- 运行记录
- 轨迹
- 调试视图

### 第八步：做质量提升

- rerank
- eval
- feedback
- guardrails

## 六、为什么当前下一步应该去做“更丰富的工具选择”

因为当前系统虽然已经有：

- Agent Tools
- LangGraph
- 工具选择

但还没有做到：

`让模型在更多工具和更多分支之间做真正决策`

现在的下一步应该是：

- 增加更多工具
- 扩展更多工作流分支
- 记录更细的工具调用过程

这样系统才会继续从：

`带工作流的 RAG`

往：

`更像 ReAct 的 Agent`

推进。

## 七、一句话总结

这个项目真正教你的，不只是“怎么做一个 RAG + LangGraph 项目”，而是：

`真实的 Agent 工程，应该如何从底座一步一步进化成智能体系统`
