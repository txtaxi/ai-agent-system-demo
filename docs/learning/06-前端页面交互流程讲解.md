# 前端页面交互流程讲解

## 文档目标

前端逐文件精读已经解释了“每个页面做什么”。

这份文档继续往前一步，专门讲：

`用户在页面上点一下按钮之后，页面内部到底发生了什么`

也就是前端的交互流程。

这样你不只是知道页面有哪些代码，而是知道：

- 用户动作
- 页面状态
- API 请求
- 响应处理
- 界面更新

是怎么串起来的。

---

## 1. 先记住前端交互的总模式

这个项目的前端交互几乎都遵循：

1. 用户触发操作
2. 页面更新本地状态
3. 通过 `api/client.js` 调后端
4. 拿到响应后更新页面数据
5. 页面重新渲染

页面之间最典型的差别只在于：

- 有些是普通接口请求
- 有些是流式 SSE
- 有些会跳转到另一个页面

## 3. 知识库页交互流程

文件：
[KnowledgeView.vue](D:\Study\Agent\StudyPlan\enterprise-ai-agent-system\frontend\src\views\KnowledgeView.vue)

这是第一个真正的业务页。

### 3.1 页面初始化

页面加载时通常会做两件事：

1. 拉知识库列表
2. 如果已经有知识库，默认选中一个并加载文档

所以用户一打开知识库页，就能看到当前系统里已有的知识库和文档。

### 3.2 创建知识库

用户动作：

- 输入知识库名称
- 点击创建按钮

页面内部流程：

1. 读取输入框内容
2. 调用 `POST /knowledge-bases`
3. 创建成功后刷新知识库列表
4. 更新当前选中的知识库

用户视角上看到的是：

- 新知识库立刻出现在左侧或列表里

### 3.3 上传文档

用户动作：

- 选择文件
- 点击上传

页面内部流程：

1. 检查当前是否已选知识库
2. 构造 `FormData`
3. 调用 `POST /documents/upload`
4. 上传成功后刷新文档列表

### 3.4 处理文档

用户动作：

- 点击某个文档的“处理”

页面内部流程：

1. 调用 `POST /documents/{id}/process`
2. 后端完成处理后返回状态
3. 前端刷新文档列表

用户看到的是：

- 文档状态从 `uploaded` 变成 `processed`

### 3.5 查看分块

用户动作：

- 点击查看某个文档的分块

页面内部流程：

1. 调用 `GET /documents/{id}/chunks`
2. 页面把返回的 chunk 渲染到分块区域

这一步对学习很重要，因为它让你能直接观察：

`RAG 的输入材料长什么样`

## 4. 对话页交互流程

文件：
[ChatView.vue](D:\Study\Agent\StudyPlan\enterprise-ai-agent-system\frontend\src\views\ChatView.vue)

这是整个前端最复杂的页面。

## 4.1 页面初始化

页面打开时，通常先做：

1. 加载会话列表
2. 加载知识库列表
3. 如果有会话，默认选中一个
4. 拉该会话的消息

所以对话页一打开，用户就能看到历史会话和消息。

## 4.2 新建会话

用户动作：

- 点击“新建会话”

页面内部流程：

1. 清空当前消息列表
2. 清空当前 active conversation
3. 等用户下次真正发问时，再让后端创建新会话

这是一种很常见的交互方式：

`会话在真正发生第一条消息时才落库`

## 4.3 发送问题

用户动作：

- 输入问题
- 点击发送

页面内部流程：

1. 前端先把用户消息插入本地消息列表
2. 优先调用流式接口 `POST /chat/ask/stream`
3. 如果流式失败，再 fallback 到普通 `POST /chat/ask`

### 流式路径

如果流式成功：

1. 收到 `meta` 事件
   - 更新会话 ID
   - 记录 route、tool、命中知识库、agent_run_id

2. 收到多个 `chunk` 事件
   - 不断把回答内容拼接到当前 assistant 消息

3. 收到 `done` 事件
   - 写入 citations
   - 更新最终状态

### 普通路径

如果流式失败：

1. 调用普通问答接口
2. 一次性拿到完整回答
3. 更新 assistant 消息

### 这一段的学习重点

这里体现了：

- 本地乐观更新
- 流式渲染
- Fallback 机制
- 消息状态与后端响应的映射

## 4.4 展示 Agent 信息

回答返回后，对话页不仅显示答案，还会显示：

- 路由类型
- 路由原因
- 选中的工具
- 工具原因
- 命中的知识库
- 命中原因
- 是否需要人工审核

这部分交互说明：

`这个前端不是单纯聊天框，而是 Agent 执行结果的可解释展示层`

## 4.5 跳转到运行记录

用户动作：

- 点击“查看本次运行记录”

页面内部流程：

1. 从当前消息里取 `agentRunId`
2. 跳转到 `/runs?agent_run_id=...`

这条交互链非常重要，因为它把：

`回答结果`

和

`执行过程`

连接起来了。

## 4.6 提交反馈

用户动作：

- 点击“有帮助”
- 或点击“没帮助”

页面内部流程：

1. 拿当前消息关联的 `agentRunId`
2. 调用 `POST /feedback`
3. 本地标记已反馈状态

这条链把：

`聊天交互`

连接到了：

`评测闭环`

## 5. 运行记录页交互流程

文件：
[RunsView.vue](D:\Study\Agent\StudyPlan\enterprise-ai-agent-system\frontend\src\views\RunsView.vue)

## 5.1 页面初始化

页面打开时：

1. 读取 URL 里的 `agent_run_id` 或 `conversation_id`
2. 调用 `GET /agent-runs`
3. 加载 run 列表
4. 如果 URL 指定了某条 run，则自动选中它

这使得“从聊天页跳过来时直接打开对应 run”成为可能。

## 5.2 查看 run 详情

用户动作：

- 点击某条运行记录

页面内部流程：

1. 调用 `GET /agent-runs/{id}`
2. 拿到完整详情
3. 展示：
   - 基础信息
   - 路由
   - 改写
   - 检索质量
   - 工具选择
   - 命中知识库
   - 步骤轨迹

## 5.3 审核通过 / 驳回

如果某条 run 是 `pending_review`：

用户动作：

- 点击通过
- 或驳回并填写原因

页面内部流程：

1. 调用 agent_runs 审核接口
2. 后端更新 run 状态
3. 前端刷新当前 run 详情和列表

这条交互说明前端已经不仅是“查看页”，而是：

`人工参与 Agent 流程的控制台`

`失败统计`

## 7. 页面之间是怎么互相联动的

这个项目的页面不是孤立的，它们之间已经形成了联动关系。

### 知识库页 -> 对话页

知识库页准备数据，对话页使用这些数据回答问题。

### 对话页 -> 运行记录页

从回答直接跳到本次 run。

### 对话页 -> 反馈统计页

聊天页提交反馈，统计页汇总反馈。

### 反馈统计页 -> 运行记录页

从负反馈样本跳到具体 run 做失败分析。

这说明前端已经不是单页面 Demo，而是：

`围绕 Agent 执行闭环建立起来的一组互相关联页面`

## 8. 前端交互中最值得学的点

### 1. 聊天页的状态组织

这个页面很值得学，因为它把：

- 会话状态
- 消息状态
- SSE 流状态
- 反馈状态

都组织到了一起。

### 2. 结果与过程联动

从回答跳到运行记录、从负反馈跳到运行记录，这两条交互很有工程味。

### 3. 不只展示结果，也展示解释

前端不仅展示答案，还展示：

- 路由
- 工具
- 命中知识库
- 检索质量
- 人工审核状态

这很符合 Agent 产品的展示特点。
