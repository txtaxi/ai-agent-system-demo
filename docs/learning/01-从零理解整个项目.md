# 从零理解整个项目

## 文档目的

这份文档不是简单重复 README，而是站在“学习整个项目”的角度，回答下面几个问题：

- 这个项目一开始是怎么搭起来的
- 前端和后端分别承担什么角色
- 每个主要文件夹和文件代表什么
- 应该按什么顺序阅读代码
- 怎样从这个项目里学会搭一个类似的 Agent 系统

这份文档不解释 `__pycache__`、上传文件目录、空占位目录这类低价值文件，而是聚焦真正值得学习的代码。

---

## 1. 先用一句话理解项目

这个项目是一个：

`企业知识库 Agent 系统`

它不是单纯的聊天页面，也不是只有检索的 RAG Demo，而是把下面几层能力串起来了：

1. 文档上传和知识库管理
2. 文档解析、分块、向量化
3. 检索与问答
4. 自动选库与任务路由
5. Agent Tools
6. LangGraph 工作流
7. 运行记录、反馈闭环与人工审核

---

## 2. 项目是怎么一步步构建出来的

### 阶段一：先搭“能跑起来”的骨架

最早先做的不是 Agent，而是项目骨架：

- 后端：`FastAPI`
- 前端：`Vue 3 + Vite`
- 数据库：`PostgreSQL + pgvector`
- 缓存：`Redis`
- 部署：`Docker Compose`

这一阶段的目标只有一个：

`先把前后端和基础设施连起来`

如果这一步没做稳，后面所有 Agent 功能都会建在沙地上。

### 阶段二：补知识库主链路

有了骨架之后，才开始做知识库：

- 创建知识库
- 上传文档
- 保存文件
- 解析内容
- 文本切分
- 生成向量
- 存分块

这一阶段把“企业知识”真正放进了系统。

### 阶段三：把知识库变成 RAG 问答系统

接着补上：

- 相似度检索
- 问答接口
- 引用返回
- 会话与消息保存
- 流式输出

到这里，系统已经不是“资料库”，而是“可回答问题的知识系统”。

### 阶段四：开始做 Agent 化

在基础 RAG 上继续补：

- 自动选择知识库
- 问题类型路由
- 查询改写
- 工具选择
- 检索质量判断
- 是否需要补充上下文
- 结果重排

这一阶段的核心变化是：

`系统开始具备多步决策`

### 阶段五：用 LangGraph 管理流程

当流程越来越复杂，就不能继续把所有逻辑塞在一个 service 里。

所以引入 LangGraph，把流程显式化为状态图。

这一步把系统从：

`一堆 if/else`

推进成：

`有状态、可追踪、可分支的 Agent 工作流`

### 阶段六：补可观测性和评测闭环

最后再补：

- 运行记录
- 工作流轨迹
- 用户反馈
- 反馈统计
- 负反馈样本回放
- 人工审核建议和暂停/恢复

这一步的意义是：

`不只是能回答，还能解释和复盘为什么这样回答`

## 3. 根目录结构怎么理解

### `backend/`

后端服务代码目录。这里放：

- API 接口
- 数据模型
- 业务逻辑
- Agent 工作流
- 检索、反馈、审核等核心能力

### `frontend/`

前端工作台目录。这里放：

- 页面
- 路由
- 请求客户端
- 全局样式

### `docs/`

项目文档目录。这里放：

- 总览
- 架构
- 开发过程
- 阶段记录
- 收口总结
- 企业级扩展规划

### `docker-compose.yml`

整个项目的容器编排文件，定义：

- `backend`
- `frontend`
- `db`
- `redis`

### `.env.example`

环境变量模板。复制成 `.env` 后，项目按这份配置运行。

### `README.md`

项目入口说明，第一次打开项目时应该先读它。

### `项目计划书.md`

项目最初的规划文档，适合回头看“为什么项目会长成现在这样”。

## 4. 后端目录详细说明

## 4.1 `backend/app/main.py`

这是后端真正的入口文件。

它里面主要做三件事：

1. 创建 FastAPI 应用
2. 注册 CORS
3. 启动时初始化 pgvector 扩展和自动建表
4. 挂载总路由

阅读意义：

`你先看这里，就知道应用怎么启动、接口从哪里挂进去`

## 4.2 `backend/app/api/`

这是接口层，负责“接请求”和“回响应”。

### `backend/app/api/router.py`

把所有模块路由统一收口。

这个文件的作用很像“总路由表”，让 `main.py` 不需要知道每个子模块的细节。

### `backend/app/api/deps.py`

通常放依赖注入，例如数据库会话依赖。

### `backend/app/api/routes/health.py`

健康检查接口，用于确认服务是不是活着。

### `backend/app/api/routes/knowledge_bases.py`

知识库相关接口：

- 创建知识库
- 获取知识库列表

### `backend/app/api/routes/documents.py`

文档相关接口：

- 上传文档
- 获取文档列表
- 处理文档
- 获取文档分块

### `backend/app/api/routes/retrieval.py`

检索接口，用于直接测试检索能力。

### `backend/app/api/routes/chat.py`

问答接口：

- 普通问答
- 流式问答

### `backend/app/api/routes/conversations.py`

会话与消息接口：

- 会话列表
- 会话消息

### `backend/app/api/routes/agent_runs.py`

运行记录与人工审核接口：

- 获取运行记录
- 获取运行详情
- 审核通过 / 驳回

### `backend/app/api/routes/feedback.py`

反馈与统计接口：

- 提交反馈
- 获取反馈统计
- 获取负反馈样本

理解这层的关键是：

`routes 只做接口边界，不做重业务逻辑`

---

## 4.3 `backend/app/core/`

### `backend/app/core/config.py`

项目配置中心。

主要内容：

- 应用名、版本
- API 前缀
- PostgreSQL 连接参数
- Redis 地址
- 上传目录
- OpenAI 兼容接口参数
- 向量维度

阅读意义：

`你从这里能看出这个项目运行依赖什么环境`

---

## 4.4 `backend/app/db/`

### `backend/app/db/session.py`

数据库连接层。

主要内容：

- SQLAlchemy `engine`
- `SessionLocal`

阅读意义：

`你会知道后端是怎么拿数据库会话的`

### `backend/app/db/base.py`

数据库 Base 导出文件，给模型层使用。

---

## 4.5 `backend/app/models/`

这是数据库模型层，对应数据库表。

### `knowledge_base.py`

知识库表。

表示一个知识库实体，例如：

- 人事制度库
- 销售制度库

### `document.py`

文档表。

保存：

- 文件名
- 类型
- 状态
- 存储路径
- 错误信息

### `document_chunk.py`

文档分块表。

保存：

- 分块内容
- 分块索引
- metadata
- 向量 embedding

### `conversation.py`

会话表。

保存一组连续对话的顶层容器。

### `message.py`

消息表。

保存每条用户消息和系统回答。

### `agent_run.py`

Agent 运行记录表。

保存一次工作流执行的过程信息，例如：

- 路由
- 工具
- 检索质量
- 是否待审核
- 轨迹 JSON

### `feedback.py`

用户反馈表。

保存：

- 哪次 run 被评价
- 有帮助 / 没帮助
- 可选备注

### `mixins.py`

公共字段混入，例如 `id`、`created_at`、`updated_at` 之类。

这层的关键理解是：

`models 决定数据库里存什么`

---

## 4.6 `backend/app/schemas/`

这是接口数据模型层，也就是 Pydantic schema。

主要文件：

- `knowledge_base.py`
- `document.py`
- `retrieval.py`
- `chat.py`
- `agent_run.py`
- `feedback.py`
- `health.py`

这一层负责：

- 定义请求参数
- 定义响应结构
- 隔离数据库模型和接口模型

这层的关键理解是：

`models 管数据库，schemas 管接口输入输出`

---

## 4.7 `backend/app/services/`

这是整个项目最核心的业务层。

### `knowledge_base_service.py`

知识库增删查服务。

### `document_service.py`

文档服务，是知识库主链路的核心。

它负责：

- 保存上传文件
- 读取 txt / md / pdf / docx
- 调用分块器
- 生成 embedding
- 写入 `document_chunks`

### `embedding_service.py`

向量生成服务。

### `retrieval_service.py`

检索服务。

它把 query 转成向量，再用 pgvector 做相似度检索。

### `knowledge_base_router_service.py`

知识库路由服务。

根据问题自动选择候选知识库，而不是让用户手工选。

### `query_router_service.py`

问题路由服务。

判断当前问题属于：

- `qa`
- `summary`
- `comparison`

### `llm_service.py`

LLM 能力收口层。

负责：

- 回答生成
- 查询改写
- 改写评估
- 工具选择

它的策略是：

`规则优先，模型辅助`

### `conversation_service.py`

会话和消息服务。

负责：

- 获取 / 创建会话
- 保存消息

### `chat_service.py`

问答总服务。

它把下面几层串起来：

- 会话
- Agent 工作流
- 运行记录
- 消息保存
- 流式返回

### `agent_run_service.py`

运行记录服务。

负责：

- 创建 run
- 标记完成
- 标记失败
- 标记 pending_review
- 审核通过 / 驳回

### `feedback_service.py`

反馈服务。

负责：

- 保存反馈
- 聚合统计
- 返回负反馈样本

---

## 4.8 `backend/app/rag/`

这里放 RAG 相关代码。

### `backend/app/rag/chunking/base.py`

定义分块器抽象接口。

### `paragraph_chunker.py`

适合普通段落文本。

### `section_chunker.py`

适合标题 / 小节结构明显的文档。

### `policy_chunker.py`

适合制度、编号条款、规范类文档。

### `router.py`

分块策略路由器。

根据文档结构自动选择更合适的 chunker。

### `utils.py`

分块辅助函数。

这一层的关键理解是：

`不是所有文档都用一种切法`

---

## 4.9 `backend/app/agent/`

这部分是 Agent 的真正核心。

### `tools.py`

把工作流中会用到的能力抽成工具。

目前主要包括：

- `search_knowledge`
- `summarize_context`
- `compare_contexts`
- `build_qa_context`
- `build_citations`
- `analyze_retrieval_quality`
- `decide_need_more_context`
- `rerank_retrieved_items`
- `assess_human_review_need`

### `workflow.py`

LangGraph 工作流文件。

它定义了整个 Agent 的状态图：

- 路由
- 改写
- 评估改写
- 工具选择
- 检索
- 检索质量分析
- 是否补充上下文
- 扩展检索
- 重排
- 人工审核判断
- 上下文整理
- 生成回答

这是理解整个项目的重点文件之一。

## 7. 这个项目里最值得你学的工程点

### 1. 分层

这个项目不是把所有逻辑堆在一起，而是明确拆成：

- API 层
- Schema 层
- Model 层
- Service 层
- Agent 层
- 前端页面层

### 2. Agent 不是一个接口，而是一组步骤

这个项目的 Agent 不是“一次调用模型”，而是：

- 路由
- 改写
- 工具选择
- 检索
- 判断
- 重排
- 上下文整理
- 回答生成

### 3. LangGraph 的价值

LangGraph 在这里不是炫技，而是：

`把复杂流程变成可读、可追踪、可分支的状态机`

### 4. 可观测性

这个项目已经不仅能回答，还能记录：

- 走了什么 route
- 选了什么工具
- 命中了哪些知识库
- 检索质量怎么样
- 为什么进入人工审核

### 5. 反馈闭环

有了反馈、统计和负反馈样本回放，系统才开始具备“自我改进”的基础。

如果你要用一句话记住整个项目，可以记成：

`这是一个基于 FastAPI、Vue、PostgreSQL、pgvector 和 LangGraph 构建的企业知识库 Agent 项目，它把知识库、RAG、工具选择、工作流、运行记录、反馈闭环和人工审核整合成了一个可演示、可调试、可扩展的完整系统。`
