# 阶段06-反馈与评测闭环

## 阶段目标

为当前企业知识库 Agent 系统补上最小版反馈闭环，让系统不只是“能回答”，还能开始积累“回答是否有帮助”的信号。

本阶段目标包括：

- 保存用户对回答的正负反馈
- 将反馈与 `agent_run` 关联
- 在对话页直接提交反馈
- 为后续评测和优化打基础

## 为什么现在要做反馈

前面几个阶段已经完成：

- RAG 主链路
- 自动选库和跨库检索
- 工具选择
- 查询改写与回退
- 检索质量评估
- 重排
- HITL 建议节点
- 运行记录

到这个阶段，系统已经具备了“过程可观测”。  
下一步就要补“结果是否被用户认可”。

这就是反馈层的价值：

- 帮助后续评测
- 支持失败案例分析
- 为检索和工作流优化提供真实信号

## 本阶段设计

### 1. 新增反馈数据模型

新增表：

- `feedback`

核心字段：

- `agent_run_id`
- `conversation_id`
- `rating`
- `comment`

当前只支持两类反馈：

- `positive`
- `negative`

先把最小闭环打通，不做复杂打分体系。

### 2. 反馈与运行记录关联

本阶段没有按消息 ID 记录反馈，而是按 `agent_run_id` 记录。  
这样做的原因是：

- 当前系统已经有成熟的 `agent_runs` 体系
- 每次问答都有唯一 `agent_run_id`
- 反馈天然可以和完整执行过程关联起来

这比单纯按“消息内容”做反馈更有工程价值。

### 3. 对话页直接提交反馈

在对话页里，系统回答消息下方新增：

- `有帮助`
- `没帮助`

用户点击后会调用反馈接口，把当前回答对应的 `agent_run_id` 写入数据库。

## 当前能力边界

本阶段完成后，系统已经可以：

- 对一次问答提交正负反馈
- 将反馈和运行记录关联
- 在对话页直接操作

但还没有做：

- 反馈列表页
- 反馈统计面板
- 自动根据反馈调整策略
- 更细粒度评分维度

## 下一步建议

在当前反馈闭环落地后，最自然的两个方向是：

1. `反馈统计 / Eval 面板`
- 查看正负反馈数量
- 按知识库、路由类型、工具类型聚合

2. `基于反馈的失败案例回放`
- 把负反馈记录和运行记录结合起来
- 快速找到问题出在哪个节点

当前更建议先做第 1 个。  
因为它能让系统第一次真正具备“可观测 + 可评估”的产品感。
