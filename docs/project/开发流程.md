# 开发流程

## 目标

这个项目采用“文档先行、代码落地、记录收尾”的轻量工程流程，目标是在单人开发场景下仍然保持清晰的项目管理方式。

## 每一阶段的标准动作

### 1. 开始阶段前

必须完成：

- 新建阶段文档，放到 `docs/phases/`
- 写清楚阶段目标、范围、验收标准
- 明确本阶段不做什么，避免范围失控

### 2. 开始编码前

必须完成：

- 检查 `系统架构.md` 是否需要调整
- 检查 `数据库设计.md` 是否需要补字段或新表
- 检查 `接口规范.md` 是否需要增加新接口

### 3. 编码进行中

采用“小批次实现”的方式，而不是等全部做完再写文档。

推荐每完成一批可验证能力，就做一次同步：

- 代码实现
- 基础校验
- 更新相关设计文档
- 在 `开发日志.md` 记一条本批次完成内容

这里的“一批可验证能力”指的是：

- 新增一个接口
- 打通一条主流程
- 完成一个数据库模型组
- 完成一个页面模块

不是每改一行代码就写文档，而是每完成一个可独立说明的小结果就写文档。

### 4. 完成阶段后

必须完成：

- 更新阶段文档中的“当前结果”
- 更新 `开发日志.md`
- 如果影响外部使用，更新根目录 `README.md`

## 当前项目的文档分工

- `项目计划书.md`
  管整体计划与阶段安排
- `docs/00-项目总览.md`
  管项目边界、文档结构和阶段位置
- `docs/系统架构.md`
  管系统结构和模块关系
- `docs/数据库设计.md`
  管数据模型
- `docs/接口规范.md`
  管 API
- `docs/phases/*.md`
  管某一阶段的目标、范围、设计、实现记录
- `docs/开发日志.md`
  管每天或每批次的完成情况

## 当前执行约定

从现在开始，每推进一个阶段或一个明显的功能批次，都按下面顺序执行：

1. 先补或更新阶段文档
2. 再写代码
3. 做基础校验
4. 更新接口规范、数据库设计、开发日志

这样做的目的不是形式化，而是保证后面写简历、准备面试、演示项目时，有完整的设计和实现脉络可讲。
