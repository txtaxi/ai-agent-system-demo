# 系统架构

## 架构目标

当前系统架构围绕三个目标设计：

1. 先把知识库问答主链路打通
2. 再把系统演化成可决策、可追踪的 Agent
3. 为后续企业级扩展保留清晰边界

## 当前整体架构

```text
用户
  ->
Vue 前端
  ->
FastAPI API 层
  ->
服务层
  -> 知识库服务
  -> 文档服务
  -> 对话服务
  -> 检索服务
  -> 反馈服务
  -> 运行记录服务
  ->
Agent 层
  -> Agent Tools
  -> LangGraph Workflow
  ->
RAG 层
  -> 文档解析
  -> 多策略分块
  -> 向量化
  -> 检索
  -> 重排
  ->
数据层
  -> PostgreSQL
  -> pgvector
  -> Redis
```

## 前端结构

前端位于 `frontend/src/`，当前页面包括：

- 首页
- 知识库页
- 对话页
- 运行记录页
- 反馈统计页

前端职责：

- 展示知识库和文档处理结果
- 发起问答请求
- 展示路由、工具、知识库命中结果
- 展示运行记录与反馈统计
- 处理人工审核动作

## 后端结构

后端位于 `backend/app/`，按层次划分为：

### API 层

位于 `api/routes/`，负责：

- 知识库接口
- 文档接口
- 检索接口
- 问答接口
- 会话接口
- 运行记录接口
- 反馈接口

### 服务层

位于 `services/`，负责：

- 文档处理
- 向量化
- 检索
- 问答
- 路由
- 自动选库
- 运行记录
- 反馈统计

### RAG 层

位于 `rag/chunking/` 与相关服务中，负责：

- 多策略分块
- 文档切分
- 向量化
- 检索与重排

### Agent 层

位于 `agent/`，负责：

- Agent Tools
- LangGraph 工作流
- 节点编排
- 条件分支
- HITL 决策

### 数据层

位于 `models/` 与数据库中，负责：

- 知识库数据
- 文档与分块数据
- 会话与消息数据
- 运行记录数据
- 反馈数据

## 当前 Agent 工作流

当前 LangGraph 工作流核心节点包括：

- `route_question`
- `rewrite_query`
- `assess_rewrite`
- `tool_select`
- `search_knowledge`
- `analyze_retrieval`
- `decide_more_context`
- `expand_search`
- `rerank_results`
- `assess_human_review`
- `prepare_*_context`
- `generate_answer`

这意味着系统已经具备：

- 路径决策
- 工具决策
- 检索质量判断
- 条件回环
- 审核分支

## 运行记录与质量闭环

当前系统不是黑盒，已经具备：

- `agent_runs` 运行记录
- 工作流轨迹
- 正负反馈
- 反馈统计
- 负反馈样本回放

这层能力使系统可以：

- 调试
- 追溯
- 失败分析
- 后续评测与优化

## 架构特点

当前架构的主要特点是：

1. 前后端职责清晰
2. RAG 层与 Agent 层有边界
3. 工作流与业务服务分离
4. 运行记录与反馈系统独立
5. 便于后续增量扩展用户、权限、审核、评测和安全能力

## 当前结论

当前架构已经足以支撑：

- 一个完整 Agent 项目
- 求职展示
- 后续企业级扩展

这里的“支撑后续企业级扩展”指的是架构边界清晰、便于继续补齐能力，
不代表当前版本已经具备完整的生产级治理能力。

后续如果进入企业化阶段，最主要的变化应是新增：

- 用户与权限层
- 审核与审计层
- Eval 与安全层
- 运维与部署层

而不是推翻当前 Agent 主体架构。
