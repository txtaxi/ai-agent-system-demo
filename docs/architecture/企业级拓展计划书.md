# 企业级拓展计划书

本文档用于指导当前 Agent 项目向更接近企业内部可落地系统的方向扩展。当前版本已经完成 Agent 主体，后续工作重点应转向企业化能力，而不是继续无边界增加实验性功能。

## 一、拓展目标

在当前版本基础上，将系统扩展为：

- 有用户和权限边界
- 有审核与审计闭环
- 有固定 Eval 能力
- 有安全与风险控制
- 有更正式部署与运维方案

## 二、拓展原则

### 1. 不推翻当前 Agent 主体

保留现有：

- RAG
- 自动选库
- Agent Tools
- LangGraph
- 运行记录
- 反馈闭环
- HITL

### 2. 企业化能力以增量方式接入

重点做：

- 新模型
- 新服务
- 新 API
- 前端管理页

避免大规模重写当前主链路。

### 3. 先做边界，再做高级体验

优先补：

- 登录
- 权限
- 审核流
- 审计
- Eval

而不是先做复杂 UI。

## 三、推荐阶段

## 阶段 A：用户与权限

目标：

- 系统从单机演示走向多用户使用

范围：

- 用户登录
- 用户角色
- 知识库归属
- 运行记录归属
- 审核权限

交付结果：

- 不同用户可以登录
- 不同角色拥有不同操作权限
- 知识库与运行记录不再是全局裸露

## 阶段 B：审核台与审计

目标：

- 把现有 HITL 从基础功能升级成企业审核流

范围：

- 待审核列表
- 审核人信息
- 审核历史
- 审核备注
- 操作审计日志

交付结果：

- 审核工作从“单个运行记录按钮”升级成“审核流程”

## 阶段 C：Eval 与质量保障

目标：

- 系统质量从“靠感觉”升级为“可度量”

范围：

- 固定问题集
- 自动跑评测
- 版本前后对比
- 失败归因统计

交付结果：

- 每次调整后都能量化评估

## 阶段 D：安全与风险控制

目标：

- 降低企业内部使用风险

范围：

- Prompt Injection 基础防护
- 敏感问答拦截
- 风险等级识别
- 更严格人工审核条件

交付结果：

- 系统对高风险场景的处理更可信

## 阶段 E：部署与运维

目标：

- 具备更正式的内部部署能力

范围：

- 生产配置
- 对象存储
- 任务队列
- 监控与告警
- 错误追踪
- 自动化初始化与部署文档

交付结果：

- 系统更接近真实内部应用运维方式

## 四、建议时间线

### 第 1 周到第 2 周

- 用户登录
- 角色与权限
- 基础鉴权中间件

### 第 3 周到第 4 周

- 审核台
- 审核历史
- 审计日志

### 第 5 周

- 固定 Eval 集
- 自动评测脚本

### 第 6 周

- 风险控制与安全规则

### 第 7 周到第 8 周

- 运维与部署增强
- 正式文档与演示环境

## 五、推荐优先级

如果时间有限，优先级建议如下：

1. 用户与权限
2. 审核台与审计
3. Eval 与回归
4. 安全与风险控制
5. 运维与监控

## 六、对当前代码的影响评估

后续扩展时，主要会影响：

- `models/`
  增加用户、角色、审计相关模型

- `services/`
  增加鉴权、审核、审计、评测、安全服务

- `api/routes/`
  新增认证、审核台、评测接口

- `api/deps.py`
  扩展权限依赖

- `frontend/src/views/`
  新增登录页、审核页、管理页

整体方式应以“新增为主、局部修改为辅”，而不是推翻式重构。

## 七、结论

当前项目已经完成 Agent 主体，后续企业级拓展不再是“继续补 Demo”，而是进入新的工程阶段。

后续扩展最合理的做法是：

- 以当前版本为主干
- 新开企业化阶段
- 增量扩展权限、审核、评测、安全、运维能力

这样既能保留当前项目成果，也能平滑演化出更接近企业内部落地的系统。
