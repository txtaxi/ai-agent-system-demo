# 企业级扩展架构

本文档用于说明：在当前 Agent 项目基础上，如果继续扩展为更接近企业内部可落地的系统，推荐采用什么架构，以及当前架构应如何平滑演化。

## 设计目标

企业级扩展阶段的目标不再只是“能跑”，而是：

- 多用户可用
- 权限边界清晰
- 审核与审计可追溯
- 质量可评估
- 运维可监控
- 扩展可持续

## 当前架构回顾

当前项目已经具备：

- 前端：Vue 3 + Vite
- 后端：FastAPI
- 存储：PostgreSQL + pgvector + Redis
- 检索：多策略分块 + 向量检索 + 路由 + 重排
- 智能体：Agent Tools + LangGraph
- 观测：agent_runs + feedback + negative samples
- 人工审核：pending_review + review

这意味着当前已经有较好的企业化扩展基础。

## 企业级扩展后的目标架构

```text
用户端 / 管理端
  ->
前端应用（Vue）
  ->
API Gateway / FastAPI 应用层
  ->
鉴权与权限层
  ->
业务服务层
  -> 知识库服务
  -> 对话服务
  -> 审核服务
  -> 反馈与评测服务
  -> 审计服务
  ->
Agent 编排层
  -> LangGraph Workflow
  -> Agent Tools
  -> HITL 节点
  ->
检索层
  -> 文档处理
  -> 多策略分块
  -> 向量检索
  -> 重排
  ->
数据层
  -> PostgreSQL
  -> pgvector
  -> Redis
  -> 对象存储（后续）
```

## 推荐新增模块

### 1. 用户与权限层

新增能力：

- 用户登录
- 用户角色
- 知识库归属
- 审核权限
- 页面级与接口级权限校验

推荐模块：

- `models/user.py`
- `models/role.py`
- `services/auth_service.py`
- `api/deps.py` 扩展鉴权依赖

### 2. 审核与审计层

新增能力：

- 待审核队列
- 审核人
- 审核历史
- 操作审计日志
- 审核备注和结果追踪

推荐模块：

- `models/audit_log.py`
- `services/audit_service.py`
- `services/review_service.py`

### 3. Eval 与质量层

新增能力：

- 固定 Eval 数据集
- 自动评测脚本
- 版本前后效果对比
- 问题分类统计
- 失败归因分析

推荐模块：

- `evals/`
- `services/eval_service.py`
- `api/routes/evals.py`

### 4. 安全与风险控制层

新增能力：

- Prompt Injection 基础防护
- 高风险问题识别
- 外部工具白名单
- 敏感输出拦截
- 强制审核规则

推荐模块：

- `services/guardrail_service.py`
- `services/risk_service.py`

### 5. 运维与部署层

新增能力：

- 对象存储
- 后台任务队列
- 监控与告警
- 正式日志与错误追踪
- 更清晰的生产环境配置

推荐模块：

- Celery / RQ 或后台任务体系
- 对象存储服务接入
- Prometheus / Grafana / Sentry 等后续方案

## 当前架构如何平滑演化

当前结构已经有以下优点：

- 模型层、服务层、API 层已分离
- Agent 工作流与业务服务有边界
- 反馈与运行记录已经独立
- 前端页面已经按业务分开

因此企业级扩展时，主要是：

- 新增模块
- 扩展模型字段
- 在 API 上加鉴权和权限依赖
- 对运行记录、反馈和审核引入操作者信息

而不是推翻式重构。

## 最终建议

扩展企业级版本时，应坚持：

1. 保持当前 Agent 主体不乱动
2. 以鉴权、审核、评测、安全、运维为主要新增层
3. 采用增量演进，不推翻现有工作流和数据层
